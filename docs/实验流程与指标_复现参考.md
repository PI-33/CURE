# CURE 实验流程、产出指标与论文复现对照

本文档基于当前代码库（`run.py` → `optimization/*` → `evaluation/eval.py`）梳理：**一次完整训练实验会跑哪些阶段、写出哪些文件、各指标含义是什么**，便于对照论文（NeurIPS 2025 Spotlight, arXiv:2506.03136）做复现与对齐。

> **路径约定**：若已设置环境变量 `CURE_EXPERIMENT_DIR`（由 `run.py` 自动设为 `experiments/<exp_name>` 的绝对路径），则下文 `<exp>` 表示该目录；否则部分脚本会回退到 `optimization/temp_data` 等旧路径。推荐始终通过 `python run.py` 启动，以保证实验隔离。

---

## 1. 总览：一次实验的时间线

入口：`项目根目录` 下执行 `python run.py`（需先配置 `optimization/optimization_config.py`）。

| 阶段 | 脚本 | 作用 |
|------|------|------|
| 环境准备 | `run.py` | 创建 `<exp>/`，设置 `CURE_EXPERIMENT_DIR`，复制 `config_snapshot.py` |
| 采样 | `optimization/sample.py` | vLLM 对训练集题目生成 `k_code` 份代码 + `k_case` 份生成单测 |
| 执行 | `optimization/execute.py` | 沙箱执行：代码×GT 私测、代码×生成单测，得到 bool 表与 BoN 统计 |
| 奖励 | `optimization/reward.py` | 按 `reward_mode` 计算 reward，写出 RL 训练 JSON |
| 步诊断归档 | `run.py` → `archive_step_summary` | 合并采样/执行/reward 指标到 `optimization/diagnostics/step_*.json` |
| 训练 | `optimization/train.py`（`python -m train`） | Ray + DeepSpeed PPO，读 RL JSON，更新权重到 `<exp>/ckpt/<optimized_model_name>/` |
| 周期性评测 | `evaluation/eval.py` | 在 `eval_dataset` 上算与训练同类的指标（独立 temp/results） |
| 周期性存档 | `run.py` | 每 `save_interval` 步拷贝 ckpt 到 `optimization/ckpt/iter<步号>/` |

**步数与 `step_*.json` 编号**（便于对齐曲线）：

- `i=0`：用**基座** `pretrained_model` 做 sample → execute → reward → `archive_step_summary(0)` → train（第 1 次参数更新）。
- `i=1 … total_steps-1`：用**当前优化模型**重复上述管线，并 `archive_step_summary(i)`。
- `i=total_steps`：只做 **evaluation**（及满足条件时的 **save**），**不再** sample/reward/train；因此诊断文件为 `step_0.json` … `step_{total_steps-1}.json`，共 `total_steps` 个与 RL 步对齐的摘要（若 `total_steps=350`，则最后一条训练步摘要为 `step_349.json`，第 350 次循环仅评测）。

## 2. 目录与文件清单（按类型）

以下路径均相对于 `<exp>/`（即 `experiments/<exp_name>/`）。

### 2.1 配置与快照

| 路径 | 说明 |
|------|------|
| `config_snapshot.py` | 训练开始时从 `optimization/optimization_config.py` 复制，**复现时以该文件为准**还原超参。 |

### 2.2 训练管线中间数据（每步覆盖/更新）

| 路径 | 写入方 | 说明 |
|------|--------|------|
| `temp_data/outputs-rl-<sanitized_model>-<train_dataset>.json` | `sample.py` 初始写入；`execute.py` 回填执行结果 | 每题含生成代码/单测、prompt、长度、执行矩阵等；体积大，**一步一份当前 rollout**。 |
| `temp_data/last_step_sample_metrics.json` | `sample.py` | 本步采样统计（见 §3.1）。 |
| `temp_data/last_step_execute_metrics.json` | `execute.py` | 本步执行与 BoN 统计（见 §3.2）。 |
| `temp_data/reward_diagnostics.json` | `reward.py` | 本步 reward 诊断；可含 `per_problem`（见 §3.3）。 |
| `temp_data/rl_data.json` 或 `rl_code_data.json` + `rl_case_data.json` | `reward.py` | `separate_training=True`（默认）时拆成 code/case 两个 JSON，供 PPO 读取。 |

`<sanitized_model>`：模型路径中 `/` 替换为 `.`（例如 `/path/to/Qwen/7B` → `..path.to.Qwen.7B`）。

### 2.3 文本结果（追加写入，便于 grep / 脚本解析）

目录：`optimization/results/`。

| 文件名模式 | 写入方 | 内容 |
|------------|--------|------|
| `results-rl-<sanitized_model>-<train_dataset>.txt` | `sample.py`、`execute.py`、`reward.py` 按顺序**追加** | 每步多行：响应长度 → 各类 acc / p_01 / BoN → reward 统计。 |
| `results-<sanitized_model>-<train_dataset>.txt` | 同上（基座模型路径时） | 与上类似；若基座与优化后路径 sanitize 后不同，会各有一份。 |

训练过程中实际写入的模型路径：`pretrained_model`（第一步）与 `<exp>/ckpt/<optimized_model_name>`（后续步），故通常存在**两套** `results-*` 前缀（基座 + optimized）。

### 2.4 步级诊断 JSON（轻量、适合画曲线）

| 路径 | 说明 |
|------|------|
| `optimization/diagnostics/metrics_field_help_zh.json` | 各字段中文说明（首次生成）。 |
| `optimization/diagnostics/step_<i>.json` | 合并 `last_step_sample_metrics`、`last_step_execute_metrics`、`reward_diagnostics`（**剔除** `per_problem` 及 `gt_correlation.all_values` 以控制体积）。 |

### 2.5 训练过程监控

| 路径 | 说明 |
|------|------|
| `optimization/tb_logs/` | TensorBoard：`train/*`（loss、KL、entropy 等）、`experience/*`（advantage、group reward 等），`step_key` 为 `global_step`（与 `run.py` 传入的 `--step` 一致）。 |

### 2.6 权重与备份

| 路径 | 说明 |
|------|------|
| `ckpt/<optimized_model_name>/` | 每步训练结束后的**当前**可加载模型目录（HF 格式）。 |
| `optimization/ckpt/iter<i>/` | 每 `save_interval` 整目录拷贝。 |
| `optimization/ckpt/final/` | 全部循环结束后的拷贝。 |
| `train.log` | 若环境变量 `CURE_LOG_FILE` 指向外部日志，结束时链到该文件。 |

### 2.7 训练期评测（Hold-out）

目录：`evaluation/temp_data/`、`evaluation/results/`（均在 `<exp>` 下）。

| 路径 | 说明 |
|------|------|
| `evaluation/temp_data/outputs-eval-<sanitized_model>-<eval_dataset>.json` | 评测集上的生成与执行明细。 |
| `evaluation/results/results-eval-<sanitized_model>-<eval_dataset>.txt` | 追加写入评测指标（文案略长于 optimization 版，含英文解释句）。 |

触发条件：`run.py` 中 `i % eval_interval == 0`（`i` 从 1 到 `total_steps`）。**默认**入口里第一行对基座的 `evaluation(...)` 被注释掉，若要对齐「训练前 zero-shot 评测」，需自行打开。

### 2.8 独立终评（可选）

在 `evaluation/eval.py` 中若 `is_final_eval=True`，结果写入  
`evaluation/results/results-eval-<model>-final_eval.txt`，与训练内周期性评测文件名不同。

## 3. 指标定义与数据字段

### 3.1 采样（`last_step_sample_metrics.json` / `results-*.txt` 首段）

| 字段 / 日志行 | 含义 |
|---------------|------|
| `mean_code_response_tokens` | 本步所有生成代码 completion 的平均 token 数（与 tokenizer 编码一致）。 |
| `mean_case_response_tokens` | 本步所有生成单测相关 completion 的平均 token 数。 |
| `code response length: ...` | 同上 mean_code，文本日志。 |
| `case response length: ...` | 同上 mean_case。 |

**论文复现**：温度、top-p、`k_code`/`k_case`、`n_sample_per_step`、prompt 模板均在 `optimization_config.py`（与论文 Appendix 描述对齐处需逐条核对）。

### 3.2 执行（`last_step_execute_metrics.json` / `execute.py` 追加段）

在**训练集当前 batch** 上，对每道题：

- 用 **GT 私测**（`test_input`/`test_output`，数量受 `max_ground_truth_test` 等配置限制）判定每份代码是否全对，得到 **code** 侧准确率。
- 仅在「存在至少一份对 GT 全对的代码」的题目上，看生成单测能否与这些好代码一致执行，得到 **case** 侧指标。
- **BoN**：对每个 `scale_tuple_list` 中的 `(num_code, num_gen_tests)`，在前 `num_code` 份代码、前 `num_gen_tests` 份生成单测组成的子矩阵上，按「生成单测上通过数最多的那条代码」选股，再看该代码在 **GT 私测** 上是否全过；统计题目级 `acc` 与测试级 `accumulate_acc`。

| 字段 | 含义 |
|------|------|
| `code_acc` | 题目维度：至少有一份生成代码在**全部所用 GT 私测**上通过的题目占比。 |
| `code_accumulate_acc` | 所有「代码 × GT 测试」对上，输出与期望一致的比例。 |
| `case_acc` | 条件于「存在全对 GT 的代码」的题目：生成单测在该题上「与所有好代码一致」的题目级比例。 |
| `case_accumulate_acc` | 好代码 × 生成单测 交叉上执行一致的比例。 |
| `p_01_as_logged` | 日志打印为 `p_01:`；**数值上等于 `1 - p_01_internal`**，其中 internal 度量错误代码在「被判为好的生成单测」上仍通过的比例（与论文中区分度/误判相关统计对应，细节见代码聚合）。 |
| `p_00` | 错误代码 × 错误单测 组合上仍判通过的比例。 |
| `bon_by_scale` | 列表，每项：`num_code`, `num_gen_tests`, `acc`, `accumulate_acc`。 |

**论文复现**：`scale_tuple_list` 常用 `[(4,4),(16,16)]` 与 Table/Figure 中 N=M=16 及中间尺度一致；`num_chunks` 需与 `k_code*k_case*n_sample_per_step` 成比例以免执行过慢或卡住。

### 3.3 奖励（`reward_diagnostics.json` / `reward.py` 追加段）

| 字段 | 含义 |
|------|------|
| `reward_mode` | `gt_based` / `self_bootstrap` / `pairwise_disagreement`（与 `optimization_config.reward_mode` 一致）。 |
| `code_reward` / `case_reward` | `mean`, `std`, `num_groups`, `num_samples`：写入 RL JSON **前**、组内归一化前的原始 reward 统计。 |
| `problems` | `total` / `used` / `skipped`：自举或低信心过滤时会 skip。 |
| `gt_correlation` | 当自举等模式可算时：与 GT-based code reward 的 Spearman 相关（**仅诊断**）；完整 `all_values` 只在 `temp_data/reward_diagnostics.json`，不会进入 `step_*.json`。 |
| `per_problem` | 每题状态、可选 rollout 调试块；可在 config 中打开 `reward_diagnostics_include_rollouts`（**文件会很大**）。 |
| `rollout_debug_meta` | 是否嵌入 rollout、截断长度等元信息。 |

文本日志中 `self_bootstrap_stats` / `gt_correlation` 行在 `self_bootstrap` 或 `pairwise_disagreement` 下都会按统一格式打印，便于日志解析脚本处理。

### 3.4 PPO 训练（TensorBoard / loguru）

| 标量（前缀示例） | 含义 |
|------------------|------|
| `experience/<tag>/avg_custom_rewards` | 自定义 reward（组内相对/标准化后信号）在 experience 上的平均。 |
| `experience/<tag>/avg_response_length` | 当前 batch 响应长度相关均值。 |
| `experience/<tag>/avg_advantages`（及 `_abs`） | Advantage 均值/绝对值均值。 |
| `experience/<tag>/avg_group_raw_reward` 等 | 组内原始/相对 reward 与 std。 |
| `train/<tag>/policy_loss`, `kl_loss`, `total_loss`, `entropy`, `clip_ratio`, `actor_lr` | PPO 训练项。 |

`<tag>` 在代码路径与 `separate_training` 下对应 `code` / `case` 等数据源（以实际 `trainer` 日志为准）。

## 4. 训练期评测 `eval.py` 与训练管线的差异

- **数据**：`eval_dataset`（如 `LiveBench`），JSON 位于 `data/<dataset>.json`；需先按 `data/readme.md` 下载。
- **指标**：与 §3.2 同一套 code/case/p_01/p_00/BoN 逻辑；`single_eval=False` 时写入 BoN 与单测指标，`True` 时仅代码准确率与长度。
- **输出文件名**：`results-eval-<sanitized_model>-<eval_dataset>.txt`（非 `rl-` 前缀）。
- **配置入口**：命令行参数由 `run.py` 传入；独立跑评测时改 `evaluation/evaluation_config.py` 或 CLI。

## 5. 论文实验复现检查清单（建议顺序）

1. **数据**：`CodeContests_train` 训练；评测集至少覆盖论文报告的 LiveBench / CodeContests / LiveCodeBench / CodeForces / MBPP（见 `data/readme.md`）。
2. **基座模型**：与论文表一致（如 Qwen2.5-7B/14B-Instruct、Qwen3-4B 等），`pretrained_model` 指向本地或 HF 缓存路径。
3. **采样**：`k_code`, `k_case`, `temp`, `top_p`，`n_sample_per_step`，`max_generation_token`，`p_give_example` 与论文设定一致。
4. **执行**：`max_ground_truth_test`，`scale_tuple_list`，`num_chunks`。
5. **奖励**：`reward_mode`（原版 CURE 对应 GT-based；扩展实验含 self-bootstrap、pairwise_disagreement）；`enable_efficient` 对长 CoT 模型与论文 3.4 一致；`separate_training`。
6. **训练**：`actor_learning_rate`，`kl_loss_coef` / `use_kl_loss`，`prompt_max_len` / `generate_max_len` / `packing_max_len`，`total_steps`，GPU `gpu_groups` 与 `actor_num_gpus_per_node`。
7. **记录**：用 `optimization/diagnostics/step_*.json` 画训练过程曲线；用 `optimization/results/*.txt` 与论文 Figure 对齐；用 `evaluation/results/results-eval-*.txt` 对齐主表评测。
8. **可复现性**：保留 `<exp>/config_snapshot.py`、最终 `ckpt` 与关键 `results-*.txt`。
