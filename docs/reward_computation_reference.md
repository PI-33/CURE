# 从数据采样到奖励与训练：全流程说明

本文档按**时间顺序**说明一步强化学习迭代中：任务如何采样、模型如何生成代码与单元测试、执行如何得到布尔矩阵、奖励如何计算、如何进入策略更新。供写论文时自行组织章节与符号。  
下文用「一步」表示一次 `sample → execute → reward → train` 的完整子循环。

---

## 0. 整体循环（一步之内）

一步内逻辑顺序为：

1. **任务采样**：从训练集中随机抽取若干道编程题（数量由配置决定，且不超过数据集大小）。
2. **生成阶段**：对每道题，用当前策略模型（通常经 vLLM 加载）独立采样多份**代码**与多份**单元测试**（数量由 \(k_{\text{code}}\)、\(k_{\text{case}}\) 给定）。
3. **解析与拼接**：从模型输出中解析可执行代码片段；从测试生成输出中解析「输入 / 期望输出」字符串；与数据集中给出的**隐藏 GT 测试**一起，组成「执行阶段」所需的全部测试用例列表。
4. **执行阶段**：对**每一份代码**在**每一条测试用例**上运行（沙箱中执行、比对输出），得到通过/失败，组装成布尔矩阵。
5. **奖励阶段**：根据该矩阵（以及是否采用自举/GT 两种模式之一）计算每条代码、每条生成测试的标量奖励；再组内标准化、可选长度正则，写成策略训练用的「成组样本」。
6. **策略更新**：用上述样本对同一语言模型做一轮（或 code 与 case 分两轮）相对偏好式的策略优化。

多步训练则重复上述过程，每步模型权重更新后，下一轮采样使用的是更新后的模型。

---

## 1. 数据与任务采样

每道训练题在数据集中通常包含（与具体数据集字段一致）：

- 题目描述；
- 若干**公开**样例（`example_input` / `example_output`），可在提示中展示；
- 若干**私有**测试（`test_input` / `test_output`），用于执行阶段与 GT 判定。

**一步内**从训练集中**无放回式随机子集**抽取 \(N\) 道题（\(N\) 为配置中的「每步采样题数」），作为本步 rollouts 的任务集合。

---

## 2. 每道题在执行前的准备

### 2.1 隐藏 GT 测试的截取

对每道题，从私有测试中取前 \(t\) 条作为本步使用的 GT 测试（\(t\) 不超过配置上限与题目自带测试总数）。记：

- GT 测试在「执行列表」中的列对应前 \(t\) 个位置；
- 这些输入/输出对**不参与**模型生成，但会参与**执行**与（在 GT 模式下）**奖励**。

### 2.2 生成任务的提示构造

- **代码生成**：用题目描述与系统提示（要求用标准输入输出格式等）拼成一条用户提示；同一道题重复 **\(k_{\text{code}}\)** 次，每次独立解码，得到**\(k_{\text{code}}\)** 条完整模型输出，再从中解析出**\(k_{\text{code}}\)** 段 Python 代码。
- **单元测试生成**：用另一套「生成测试用例」的提示；若配置为以概率在提示中**附带公开样例**，则按概率决定是否把 `example_*` 写入提示。同一道题重复 **\(k_{\text{case}}\)** 次，得到 **\(k_{\text{case}}\)** 条输出，再从中解析出 **\(k_{\text{case}}\)** 组（输入字符串、期望输出字符串）。

解析失败时，对应位置会得到占位空串，后续执行中往往表现为失败。

### 2.3 解码与并行

所有「代码提示」与「测试提示」可打乱顺序后，分发给多个 GPU 上的推理引擎并行解码；解码结束后按原始顺序还原，保证第 \(i\) 份代码与第 \(j\) 份测试仍属于正确的题目。

---

## 3. 执行列表的组装

对每道题，执行阶段使用的**测试用例序列**为：

1. **先**接 GT 私有测试：\((\text{GT 输入}_1,\text{GT 输出}_1),\ldots,(\text{GT 输入}_t,\text{GT 输出}_t)\)；
2. **再**接模型生成的测试：\((\text{gen 输入}_1,\text{gen 输出}_1),\ldots,(\text{gen 输入}_{k_{\text{case}}},\text{gen 输出}_{k_{\text{case}}})\)。

总列数（测试条数）为 \(t + k_{\text{case}}\)。  
行数为 **\(k_{\text{code}}\)**，每条对应一份解析出的代码。

---

## 4. 执行与布尔矩阵

对每一对「代码 \(\times\) 测试」，在隔离环境中执行代码：将测试输入作为标准输入喂入，捕获标准输出，与期望输出做**规范化空白后**的字符串相等判定。

得到矩阵：

\[
\mathbf{B} \in \{0,1\}^{k_{\text{code}} \times (t + k_{\text{case}})}, \quad
B_{ij}=1 \text{ 表示第 } i \text{ 份代码在第 } j \text{ 条测试上通过}。
\]

- 前 \(t\) 列：GT 子矩阵，记为 \(\mathbf{B}^{\text{gt}}\)（用于 GT 奖励或诊断）。
- 后 \(k_{\text{case}}\) 列：生成测试子矩阵，记为 \(\mathbf{M}\)（自举奖励的主输入）。

若某题没有任何可执行代码或没有任何测试，则该题可能整题跳过，不产生矩阵。

---

## 5. 奖励计算（两种模式）

配置在「自举」与「GT 监督」两种模式间切换。以下分别说明。

### 5.1 公共后处理（两种模式在写出训练标签前都会做）

对**同一道题、同一类任务**（要么全是代码候选，要么全是测试候选）下的若干标量，先做：

- **组内标准化**：\(\hat{r}_k = (r_k - \mu)/\sigma\)；若方差为 0 则整组丢弃。
- **可选长度正则**：在「高效/长思维」等配置下，对奖励取符号后对正样本按响应长度做惩罚，再平衡正负量级（仅当开关打开时）。
- 最终把「同一 prompt」下的多条响应与对应标量打成**一组**，供策略梯度使用。

---

### 5.2 自举模式（不依赖 GT 列参与主奖励）

**输入**：仅使用生成测试子矩阵 \(\mathbf{M}\)（\(k_{\text{code}} \times k_{\text{case}}\)）。GT 列**不进入** code/case 主奖励公式。

**可参与性**：若 \(\mathbf{M}\) 为空，或不存在任一列使得通过率严格在 \((0,1)\) 之间（全通过或全失败列无区分信息），则**本题整题跳过**，不产生 code/case 训练组。

**EM 估计**（迭代若干轮）：联合估计

- \(q_i\)：第 \(i\) 份代码的质量分数；
- \(w_j\)：第 \(j\) 个生成测试的权重（非可区分列恒为 0）；
- \(e_j \in \{0,1\}\)：第 \(j\) 个测试的「期望行为」（好代码应通过还是不应通过）。

可用可选的 **anchor**：用公开样例在 GT 列前若干列上的通过率，对 \(q\) 的初值或迭代更新做凸组合，强度为 \(\alpha\)。

**代码（code）原始奖励**（对第 \(i\) 份代码）：

\[
r^{\text{code}}_i
= \frac{\sum_{j} w_j \cdot \mathbb{1}[M_{ij} = e_j]}{\sum_j w_j}
\]

（要求 \(\sum_j w_j > 0\)，且组内 \(r^{\text{code}}\) 有方差，否则不写入。）

**测试（case）原始奖励**（对第 \(j\) 个生成测试）：  
**仅**用 EM 得到的向量 \(q\) 与 \(\mathbf{M}\) 的第 \(j\) 列，与 \(w,e\) 无关：

- 若该列全 0 或全 1，或列上无方差 → \(r^{\text{case}}_j = 0\)。
- 否则  
  \[
  r^{\text{case}}_j = \rho\bigl(q,\, M_{1j},\ldots,M_{k_{\text{code}}j}\bigr)
  \]  
  其中 \(\rho\) 为 **Pearson 相关系数**（好代码更易通过、差代码更易失败时，相关为正）。

若 \(q\) 几乎无方差，或整题 \(r^{\text{case}}\) 几乎无方差，则本题 case 组丢弃。

**若存在 GT**（\(t>0\)），可额外计算「GT 下的 code 分数」与「自举 code 分数」的 **Spearman 相关**，仅作**诊断**，不参与训练标签。

---

### 5.3 GT 监督模式（原版 CURE 风格）

**代码（code）原始奖励**：第 \(i\) 份代码在 **全部 \(t\) 条 GT 测试**上的平均通过率：

\[
r^{\text{code}}_i = \frac{1}{t}\sum_{k=1}^{t} B^{\text{gt}}_{ik}.
\]

**测试（case）原始奖励**：依赖「是否存在 GT 全对代码」。

- 若**没有任何**代码在全部 GT 上通过 → **本题不产生 case 奖励**。
- 否则：在「GT 全对」的代码上，若某生成测试列被**全部**正确代码通过，则标为**好测试**（正号）；否则标为**坏测试**（负号）。  
  再用**错误代码**在这些列上的通过/失败模式做**缩放**：好测试上，错误代码越常被拒绝，缩放越大；坏测试上，错误代码越常误通过，缩放越大（具体形式与是否「后阶段」二值化有关）。  
  得到向量 \(r^{\text{case}} = \text{sign} \odot \text{scale}\) 后，同样做组内标准化与可选长度正则。

---

## 6. 从奖励到策略更新

训练数据按「代码任务」与「测试生成任务」两类组织：同一道题对应两条 prompt（代码生成 prompt、测试生成 prompt），各带一组响应与奖励。

若配置为**分离训练**：先用所有代码组的样本更新一次策略，再用所有测试组的样本更新一次策略（**同一套模型参数**，两轮更新）。

若配置为**合并**：则把两类样本打乱后一起更新。

在进入实际梯度计算前，训练器通常还会对**同一 prompt 下**多条响应的奖励再做一次**组内减均值、除标准差**，使优化目标表现为「相对排序」而非绝对标量。

---

## 7. 论文叙述时可强调的几条因果链

1. **采样**：任务子集 + 每题多条随机解码 → 覆盖不同代码与测试假设。  
2. **执行**：\(\mathbf{B}\) 把「语义」变成「可优化的二元事实」。  
3. **自举**：仅用 \(\mathbf{M}\) 与 EM 估计的 \(q,w,e\) 定义 code 奖励；case 奖励用 **\(q\) 与列向量的相关性** 强调「与质量排序一致」的测试。  
4. **GT 模式**：code 直接对齐 GT；case 用「正确代码是否一致通过」与「错误代码是否被区分」定义测试好坏。  
5. **联合优化**：同一模型既学写代码，也学写能区分代码的测试，通过共享参数与交替/合并的 RL 更新耦合。

## 8. 符号一览（便于与论文记号对齐）

| 符号 | 含义 |
|------|------|
| \(N\) | 一步内抽样的题目数 |
| \(k_{\text{code}}\) | 每题采样的代码条数 |
| \(k_{\text{case}}\) | 每题采样的生成测试条数 |
| \(t\) | 每题使用的 GT 私有测试条数 |
| \(\mathbf{B}\) | 全量通过矩阵，\(k_{\text{code}} \times (t + k_{\text{case}})\) |
| \(\mathbf{M}\) | 生成测试子矩阵，\(k_{\text{code}} \times k_{\text{case}}\) |
| \(q,w,e\) | 自举 EM 输出的代码质量、测试权重、测试期望行为 |
| \(r^{\text{code}}_i,\, r^{\text{case}}_j\) | 原始标量奖励（标准化前） |
