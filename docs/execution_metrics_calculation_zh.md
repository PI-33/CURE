# CURE 训练步 `execution` 指标说明

## 这些数是哪一步实验里来的

在 **CURE 主训练循环的每一步**里，会先用当前策略模型对一批训练题做 **rollout**：对每道题采样多条生成代码、多条模型生成的单元测试；再在隔离环境里 **逐条执行**「代码 × 用例」，把实际输出与期望输出比对。  
下面这些指标就是在 **该步、该批 rollout 全部执行完成之后**，对整批题目汇总得到的诊断统计，通常会记入当步的诊断 JSON 的 **`execution`** 字段（以及同内容的执行摘要文件）。

**数据设定（理解指标的前提）**

- 每道题会用到若干条 **数据集里的真实私有测试（GT）** 作为判代码是否正确的金标准。
- 同时还有 **模型自己生成的测试**；执行时用模型给出的输入跑代码，再和模型给出的期望输出比对，得到「这条代码在这条生成测试上是否一致」。
- **好代码**：在该题全部抽样到的 GT 上，每一条测试都通过的那几条生成代码。  
- **好测试**：只在与「好代码」相关的意义上定义——某一列生成测试若与 **每一道好代码** 的执行结果都一致，则称该列为好测试。  
- **坏代码 / 坏测试**：分别是「不是好代码」的行、「不是好测试」的列（在相应定义下）。

## 指标列表

### `code_acc`

- **是什么实验里算出来的**：当前训练步、当前 batch 的 rollout 执行汇总。  
- **含义**：在所有被采样的 **生成代码条数**里，有多大比例在 **本题用到的全部 GT 私有测试** 上 **条条都过**（整题上「满分代码」占全部代码样本的比例）。  
- **注意**：按 **代码条数** 加权聚合，**不是**「有多少比例的题目至少有一条满分代码」。

### `code_accumulate_acc`

- **是什么实验里算出来的**：同上一步执行汇总。  
- **含义**：把所有题目里每一个「某条生成代码 × 某一个 GT 测试」当成同等权重的格子，其中执行结果与 GT 期望一致的比例。衡量的是 **在单测粒度上** 代码相对 GT 的贴合程度，比 `code_acc` 更细（不要求整行全对）。

### `case_acc`

- **是什么实验里算出来的**：同上，但 **只统计「至少存在一条好代码」的题目**。  
- **含义**：在这些题目里，在 **所有模型生成的测试列** 中，有多大比例列属于 **好测试**（即与该题 **所有** 好代码的执行结果都一致）。用来粗看生成单测里有多少「和已知正确行为一致」的列。

### `case_accumulate_acc`

- **是什么实验里算出来的**：同上，仍只在「至少有一条好代码」的题目上统计。  
- **含义**：在 **好代码** 与 **生成测试** 的所有交叉格上，执行结果与生成测试所给期望一致的比例。衡量生成单测与「已通过 GT 的代码行为」之间的 **格级** 一致程度。

### `p_01_as_logged`

- **是什么实验里算出来的**：同上；仅当该题同时存在 **坏代码** 和 **至少一列好测试** 时才参与统计。  
- **含义**：在 **坏代码 × 好测试** 的每一个组合上，看执行结果是否仍被算成与好测试的期望一致（True）。`p_01_as_logged` 就是这些格子里 **True 的比例**，即：**明明不是 GT 意义下的好代码，却在「好测试」上仍显示为通过** 的比例，可理解为好测试对坏代码的 **放过率 / 漏检率**。（实现上对应「先算坏×好格上为 False 的比例，再用 1 减」后写入日志与 JSON。）

### `p_00`

- **是什么实验里算出来的**：同上；仅当该题同时存在 **坏代码** 和 **坏测试** 时才参与统计。  
- **含义**：在 **坏代码 × 坏测试** 的格子上，执行结果仍与坏测试期望一致（True）的比例。可理解为 **弱或错的单测** 与 **错误代码** 凑在一起时，仍出现 **虚假通过** 的比例。

### `bon_by_scale`

- **是什么实验里算出来的**：仍在同一批 rollout 执行结果上，按配置里多组 \((N, M)\) 分别再算一遍（例如 \((4,4)\) 与 \((16,16)\)）。  
- **含义（Best-of-N 风格）**：对每道题，只取前 \(N\) 条生成代码和前 \(M\) 条生成测试；**不能看 GT 来选码**，只能在 \(N\) 条里选「在 \(M\) 条生成测试上通过个数最多」的那条代码（平局时取约定顺序中最前的一条）。选定后再用 **GT** 评判这条代码。  
  - 子字段 **`acc`**：各题上「选中代码在 **全部 GT** 上全对」的题目比例。  
  - 子字段 **`accumulate_acc`**：选中代码在 **各 GT 测试** 上通过的格子的总体比例（逐测平均意义上的通过率）。  
- **直观**：模拟「用生成单测当打分信号做 BoN 选码」时，真实能力（以 GT 为准）有多好；\(N,M\) 越大，通常选码信息越足，可与较小 \((N,M)\) 对比 scaling 行为。

---

## 小结

上述指标都描述 **当前训练步、当前 rollout 批** 在执行与比对之后的行为质量：前两项偏 **代码 vs GT**，中间两项偏 **生成单测 vs 好代码**，`p_01_as_logged` / `p_00` 偏 **判别可靠性**，`bon_by_scale` 偏 **不偷看 GT 的选码 + GT 验收** 的效果。
